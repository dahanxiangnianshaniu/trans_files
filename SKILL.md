# 红线条目 11.1.1-1+2 隐私声明与个人数据说明合规检查 + 11.4.1-1 留存期合规性检查

## 触发条件

当用户提及以下任一内容时触发本skill：
- 隐私声明合规检查、隐私政策检查、隐私声明审查
- 个人数据说明检查、个人数据清单检查、个人数据说明审查
- 11.1.1-1、11.1.1-2、RL-11_1_1、红线条目11.1.1
- 11.4.1-1、RL-11_4_1-1、红线条目11.4.1
- 数据控制者合规、设备供应者合规
- 隐私声明4要素、个人数据处理方式、数据存留期限
- 个人数据留存期、留存期合规、数据留存合规、留存期最小化
- 个人数据留存规范、留存期上限
- 删除个人数据能力、数据销毁能力、数据匿名化能力

## 规则解读

**11.1.1-1（数据控制者）**：隐私声明必须全面描述系统涉及的个人数据，包含4项要素：类型描述、目的、处理方式、存留期限。

**11.1.1-2（设备供应者）**：产品资料中必须提供个人数据说明，内容需包含4项要素，格式需满足公司要求（分类结构、边界清晰、数据来源标注、版本信息）。

**11.4.1-1（数据控制者）**：须基于业务处理的具体目的定义个人数据的留存期，留存期不得超出《个人数据留存规范》上限；DI 清单中的留存期须与隐私声明描述一致；业务目的已达成或留存期到期的个人数据须及时处置（删除或匿名化）。

**11.4.1-1（设备供应者）**：须提供个人数据处置（删除或匿名化）能力，帮助客户满足留存期最小化原则。

**注意**：数据控制者和设备供应者的4D判定标准不同（前者检查留存期数值合规，后者检查删除能力是否存在），报告模板不同，不得混用。

## 工具依赖与降级策略

**首选**：`redline-doc-audit` MCP Server，提供5个工具：`scan_documents`、`read_document`、`search_in_document`、`extract_tables`、`list_sections`。MCP Server不含LLM调用，所有分析由Agent完成。

**降级**：MCP不可用时自动切换到内置工具（Glob/Grep/Read/Bash+Python脚本），不中断流程。降级替代方案和Python脚本模板见 `references/degradation-guide.md`。仅在报告附录标注"本次检测使用降级模式"。

### 阶段一：文档发现（渐进式搜索）

用3轮搜索逐步扩大范围：

**第1轮 — 精确搜索(precise)**
- 模式一关键词：隐私声明、隐私政策、Privacy Policy、Privacy Statement、个人数据清单、个人数据列表、DI清单、personal data list、个人数据留存规范、留存规范、retention standard
- 模式二关键词：个人数据说明、个人数据清单、Personal Data Statement、Data Inventory
- 阈值：评分 ≥ 50

**数据控制者模式第1轮追加Glob扫描模式**：`**/*个人数据清单*.*`、`**/*个人数据列表*.*`、`**/*DI清单*.*`、`**/*personal*data*list*.*`、`**/*留存规范*.*`、`**/*retention*standard*.*`

**注意**：3轮搜索后的中止判断仅基于核心文档（数据控制者：隐私声明；设备供应者：个人数据说明）。DI清单和留存规范属于补充文档，未找到时不触发整体中止，其对应4D步骤标记⊘。设备供应者模式下不追加DI清单和留存规范的文件名搜索关键词，因其DI清单等同于个人数据说明，已被现有逻辑覆盖。

**第2轮 — 扩展搜索(broad)**
- 模式一关键词：隐私、声明、政策、个人信息、用户信息、数据收集、数据说明、privacy、data、personal、information、collect、statement、留存、retention、保存期限
- 模式二关键词：隐私、声明、政策、个人信息、用户信息、数据收集、数据说明、privacy、data、personal、information、collect、statement
- 阈值：评分 ≥ 30

**第3轮 — 全量扫描(fullscan)**
- 对所有文档读前200字内容预览，由Agent判断是否相关
- 目标：发现嵌入在其他文档中的相关章节

3轮搜索后仍未找到 → 结论为文档缺失（审计结论**FAIL**），跳过阶段二至五，直接进入阶段六生成报告。报告中说明3轮搜索未发现并列出搜索范围和关键词。

降级模式下的3轮搜索实现方式见 `references/degradation-guide.md`。

### 阶段二：文档确认

**模式一确认（隐私声明）**：
- 确认：标题含"隐私声明""隐私政策""Privacy Policy"，或内容描述了个人数据收集行为
- 排除：纯产品说明书、安装指南、技术规格、仅提到"隐私"但未描述数据收集

**模式二确认（个人数据说明）**：
- 确认：标题含"个人数据说明""个人数据清单"，或文档中有列出个人数据类型的结构化内容
- 排除：隐私声明（属模式一）、纯功能描述、技术架构文档

**留存规范确认（仅数据控制者模式）**：
- 确认：标题含"留存规范""留存标准""retention standard"，或文档含按数据类型定义留存期上限的表格
- 排除：纯数据清单（无上限定义）、技术规格文档

**DI清单确认（仅数据控制者模式）**：
- 确认：标题含"DI清单""个人数据清单""个人数据列表""Data Inventory"，或文档含结构化表格列出个人数据及留存期列
- 排除：纯功能描述、技术架构文档
- 注意：对设备供应者模式，DI清单即"个人数据说明"，已在现有逻辑中确认，无需重复确认

**多文件处理规则**（对DI清单和留存规范执行）：

| 扫描结果 | 处理方式 |
|---------|---------|
| 0个 | 对应4D步骤标记⊘（无法检查），标 WARNING |
| 1个 | 直接使用 |
| 多个 | 标记⊘（"存在多个XX文件，无法确定使用哪个"），标 WARNING |

隐私声明的多文件处理复用现有11.1.1-1+2逻辑，但在4D-3一致性比对时，如果隐私声明有多个，也标记⊘。

### 阶段三：内容提取

提取时必须保留四要素列的完整原始文本，不得只提取结构化分类——后续检查基于每个单元格的实际文本内容判定，而非仅判断是否为空。

**多页签/多表格完整提取**（最常见的遗漏源）：

很多xlsx/docx文档包含多个页签或多个表格，每个页签都可能包含个人数据说明数据。提取时必须：
1. 列出文档中所有页签/表格名称和行数
2. 对每个含数据的页签/表格逐一提取，不得只读第一个就停止
3. 在报告"2.1 数据来源结构"中按页签/表格分行列出提取内容
4. 总数据行数 = 所有页签/表格的数据行数之和，而非估算

> eSight等产品文档常有"eSight"和"管理面"两个页签，仅读第一个会遗漏30%+的数据。

**合并单元格拆分**：xlsx文档中场景列常有合并单元格——同一场景跨多行（如"远程通知"下有3种子场景），各子场景有不同数据项、目的、处理方式和存留策略，不得合并为一条。判断方法：如果某行"收集的个人数据项"列与上一行不同，则是独立行数据。

**隐含数据收集的章节**（除第一节外还需检查）：

- Cookie/同类技术章节：Cookie收集的数据也是个人数据
- 第三方SDK/嵌入内容：第三方服务收集的数据应纳入检查
- 自动收集的技术数据：确认文档是否有专门说明
- 第三方共享/披露章节：共享数据类型必须纳入处理方式检查——共享是处理方式的重要组成部分

识别方法：先用list_sections获取全部章节标题，逐章检查是否涉及数据收集。不得只看第一节就停止。

**第三方SDK关联**（模式一特有）：隐私声明中"三方SDK"章节的表格（第三方名称/处理目的/涉及个人信息种类/隐私声明链接），其中"涉及个人信息种类"必须关联到对应数据类型的处理方式——如果某数据类型被共享给第三方SDK，其处理方式判定应包含"共享至XX"。

**根据文档格式选择提取策略**：

- 含表格文档（.xlsx/.docx/.pdf/.chm/.html/.md）：先提取表格，检查表头是否含"数据类型/信息类型"等关键词 → 找到目标表格从行提取 → 未找到用关键词定位段落
- 纯文本文档（.txt/.rst等）：先获取章节结构定位相关章节 → 搜索关键词 → Agent从段落提取

**xlsx填写说明行过滤**：很多xlsx个人数据说明文档的数据表格中，表头下方有一行"填写说明"（内容以"填写说明："开头，描述各列应如何填写，如"填写说明：个人数据使用的场景..."），这是表头辅助信息而非实际数据行。提取时必须跳过"填写说明"行，数据行从第一个非说明、非空的实际场景行开始计数。典型位置：R5行左右。未过滤会导致数据行数虚增1-2行。

**xlsx多列列头映射**：很多xlsx文档列数>8（如OceanProtect有24列），四要素的信息分散在多列中。提取前必须先识别列头关键词，将xlsx多列映射到四要素：

| 四要素 | xlsx中对应的列头关键词 |
|--------|----------------------|
| 类型描述 | "收集的个人数据项""个人数据类型""Personal data type" |
| 目的 | "使用个人数据的目的""目的""Purposes" |
| 处理方式 | "传输方式""存储方式""存储位置""是否向第三方披露""是否跨境转移"（需合并） |
| 存留期限 | "存留期限""存留策略""存储时限""留存策略""retention" |

不得将"收集来源""来源方式"列的内容填入存留期限列——这是最常见的列错位错误。

**空值检测**：逐行逐列扫描空单元格，记录精确的空值数量和位置（行号+列名），不得用"~""约""无明显空值"等估算表述。

**DI清单留存期列提取（仅数据控制者模式）**：

在现有提取逻辑之后，增量提取以下内容：

1. **定位留存期列**：列名包含关键词"留存期""存储时长""retention""保存期限""保留期限""数据保留期"
2. **执行页签排除**：排除目录页、说明页、变更记录页等非数据页签（复用现有阶段三的页签排除逻辑）
3. **逐条提取留存期值**：对有效页签中的每条个人数据，提取其留存期值
4. **区分两种状态**：
   - "不被留存的数据"（留存期标记为"不留存""不存储""不保存"等）→ 合法状态，不算违规
   - "被留存但未定义留存期的数据"（留存期为空或未填写）→ 违规

**留存规范数据提取（仅数据控制者模式）**：

1. 读取留存规范文件，按页签提取每种数据类型的留存期上限时长
2. 形成 `{数据类型 → 上限时长}` 映射表
3. **页签排除**：同样排除目录页、说明页、变更记录页

### 阶段三B：文档结构识别

在提取内容后、检查四要素前，必须识别文档结构模式——不识别结构就直接逐项检查，极易遗漏分散在其他章节中的要素。

**快速通道**：如果提取到的表格表头同时包含以下3组关键词中的至少2组，**且文档中不存在表格之外的统一声明章节**（如含"保护/处理/存储/加密/保存"的独立正文段落，或含"留存/期限/保留/删除"的独立正文段落），直接判定为模式A，跳过三B：
- "目的/用途/收集目的"
- "处理/存储/加密/保护"
- "期限/保留/存留/销毁"

**快速通道不适用的情况**：即使表格表头含2+组关键词，如果文档存在以下任一特征，快速通道不适用，必须进入结构识别流程：
- 表格之外的正文段落包含对多个数据类型通用的声明（如"上述个人信息保存在终端本地"）
- 存在独立的统一声明章节（如Section 3有统一存留声明、Section 2有第三方SDK共享）

只有表格缺少2+项要素列时，或快速通道不适用时，才进入以下结构识别：

**模式A — 逐项自含**：每个数据类型的4项要素在同一段落/行内完整说明，四要素不依赖其他章节补充。常见于xlsx表格或四列齐全的docx表格。检查策略：逐项读取即可。

**模式B — 总-分**：第一节列数据类型，其他章节统一说明目的/处理方式/存留期限。常见于docx/htm。检查策略：跨章节关联——从数据类型列表提取类型，再从统一声明章节提取要素匹配到每个类型。

**模式C — 混合**：部分要素逐项说明，部分要素在统一章节声明。这是docx隐私声明中最常见的模式——表格有类型/目的/存留期等部分列，但处理方式或存留期限散落在正文或统一章节中。检查策略：优先使用逐项描述，缺失的要素去统一声明查找。

**识别执行方式**：读取章节标题，检查是否含独立章节：
- 含"使用/用途/目的"的独立章节 → 存在统一目的声明
- 含"保护/处理/存储/加密/保存"的独立章节 → 存在统一处理方式声明
- 含"留存/期限/保留/删除"的独立章节 → 存在统一存留期限声明

**模式识别最常见的错误——模式A误判**：叙述式文档（无表格）的各条款自含部分信息但四要素并不完整，或表格缺少"处理方式""存留期限"列而需从正文补充，这类文档**不是模式A**。模式A要求四要素在同一位置完整可得，无需跨章节关联。如果文档存在以下任一特征，即使每条数据自含部分信息，也应识别为模式B或C：
- 四要素中有任何一项分散在其他章节（如存留期限统一在第5章声明）
- 表格列数<6（缺少处理方式或存留期限列）
- 条款/表格之外的正文段落包含对多个数据类型通用的声明（如"上述个人信息保存在终端本地"）

**叙述式文档的结构识别**：纯文本叙述式隐私声明（如PAA APP的8条条款）看似逐项自含，但"卸载后删除"等统一声明分散在特定条款中，应识别为模式C——将通用条款视为统一声明章节，关联到各数据类型。

识别结果必须在报告中记录——即使是快速通道判定的模式A，也应在内容提取章节标注"文档结构识别：模式A（逐项自含）"。

### 阶段四：四要素完整性检查

根据阶段三B识别结果选择策略：模式A直接逐项判定；模式B跨章节关联；模式C优先逐项+补充统一声明。

**四要素逐项判定标准**：详见 `references/judgment-criteria.md`。该文件包含每个要素（类型描述/目的/处理方式/存留期限）的✓/△/✗判定规则、存留期限△分类详表、xlsx多列格式处理方式合并规则、第三方SDK关联规则等。

**关键约束**（不读取reference也必须遵守）：

1. **逐行审查**：对每个数据类型的每个要素，必须读取实际文本内容判定，不得只检查是否为空。最典型的错误：存留期限列非空即标✓——大量非空文本（"永久存储""长期保留"）应标△。

2. **处理方式场景化判定**（最常见的误判源）：处理方式三要素（存储方式+加密措施+是否共享）不应机械地对所有场景一刀切要求"三者齐全才✓"。不同数据使用场景对加密措施的实际需求差异巨大，必须按场景分类判定：

   | 场景类型 | 典型特征 | 加密措施要求 | 判定指引 |
   |---------|---------|------------|---------|
   | **持久化存储** | 数据写入数据库/文件系统/日志池，长期保存 | 必须说明加密措施 | 缺加密→△（如"明文存储""存储时不加密"） |
   | **端侧临时使用** | "仅在端侧使用""仅在本地处理"，不持久化 | 不强制要求加密措施 | 存储方式(端侧/内存临时)+是否共享明确→✓；缺是否共享→△ |
   | **仅读取/不保存** | "读取""获取"后不存储，如读取语言配置 | 不要求加密措施 | 明确"不保存/不存储"→✓；未明确是否保存但描述暗示临时使用→△（轻） |
   | **删除/清除操作** | 操作目的是删除数据，如"清除所有个人数据" | 不适用（操作本身是删除） | "从XX删除"→✓（删除即处理方式）；附加安全保障说明→加分项 |
   | **传输中数据** | "通过加密通道传输""TLS加密传输" | 已说明传输加密 | 传输加密+不存储+不共享→✓；传输加密但未说明存储→△（轻） |
   | **共享至第三方** | "上传到XX服务器""共享给XX" | 必须说明传输和存储端加密 | 仅说明上传未说明加密→△；两端加密+共享明确→✓ |

   **核心原则**：对"不存储/仅临时使用/仅读取/删除"类的数据，加密措施要求应放宽——这类数据在业界隐私声明中通常不要求声明加密措施，强制要求反而导致过度审计。判定时问自己：如果这是一个合规的隐私声明，读者看到"仅在端侧使用，不保存，不共享"后，还会追问"那加密了吗？"吗？对于持久化存储和第三方共享场景，读者确实会追问，此时缺加密措施标△才合理。

3. **"不存储"与"读取"的区分**（存留期限判定的关键，也是误判高发区）：
   - 原文明确写"不保存""不存储""不上传""不会将数据留在设备上"→存留期限标✓"不存储，无存留期限问题"
   - 原文写"读取""获取""收集"但未说明是否保存→**不得直接标✓**，但**也不得一律标✗**。必须区分两种情况：
     - **上下文暗示临时使用**（如"读取系统语言配置"→用于决定界面语言→逻辑上不需要存储；"获取IP地址"→用于建立连接→连接后不再需要）→存留期限标△（轻）"未明确是否保存，根据上下文推断为临时使用，但缺乏明确声明"。对应的处理方式也标△（轻）"端侧临时使用，未明确是否保存"
     - **上下文无法推断**（如"读取设备信息"但用途不明）→存留期限标△（轻）"未说明是否保存及存留期限"，而非✗——因为"读取"本身暗示数据已被获取，只是缺乏保存/销毁声明，属于"有涉及但不完整"而非"完全没有相关描述"
   - 原文写"读取"且有其他章节声明"仅临时使用不保存"→引用统一声明标✓
   - **关键原则**：存留期限标✗的条件是"完全没有相关描述"——即文档中既没有明确的时间范围，也没有策略性描述（如"卸载后删除"），也没有上下文可以推断。对于隐私声明中已列出的数据类型，既然文档承认收集了该数据，说明存留问题已经被涉及（只是回答不够明确），这类情况应标△而非✗。✗只适用于文档对该数据类型的存留问题完全没有触及的情形。
   - 不得将"读取"自动等同于"不存储"，也不得将"读取"等同于"持久化存储"——必须基于原文上下文判断

4. **四要素定义不得扩展**：四要素严格为类型描述、目的、处理方式、存留期限这4项，不得自行增加额外维度（如"数据类别""敏感等级""分类标准"等）作为✗判定依据。"类型描述"要素检查的是原文是否明确说明了收集了什么个人数据（如"姓名、电话、邮箱"），而非是否按某种分类标准标注了数据类别。如果原文未标注数据类别但已明确列出具体数据项，类型描述应标✓或△（视具体程度），不得因缺少类别标注而标✗。

5. **标✗前先查统一声明**（模式B/C）：某要素标✗之前，必须先确认该要素是否在文档的统一声明章节中有说明。如果统一声明覆盖了该数据类型，不得标✗——应引用统一声明原文并判定✓或△。

6. **模式B关联完整性**：统一声明章节通常同时包含多个要素（如第四节同时写了处理方式和存留期限）。关联时必须将统一声明中的**每一条**声明都关联到数据类型，分别判定。典型错误：只关联了存留期限，却将处理方式标✗"原文无相关描述"——这是跨章节关联不完整。

7. **四要素矩阵列结构必须完整**：矩阵必须包含以下7列：场景/数据项/类型描述/目的/处理方式/存留期限/状态，不得省略任何列。数据项过多时可截取前3-5个关键项+加"...等N项"。不得将矩阵缩减为仅"状态+存留期限"两列——完整列结构是读者理解判定依据的前提。**列名必须严格为以下7个，不得合并列、重命名列或自增列**：
   - 模式一：数据类型 / 收集的个人数据项 / 类型描述 / 目的 / 处理方式 / 存留期限 / 状态
   - 模式二：使用场景 / 收集的个人数据项 / 类型描述 / 目的 / 处理方式 / 存留期限 / 状态
   不得将"场景"与"数据项"合并为一列，不得增加"判定依据摘要"等额外列，不得将"类型描述"重命名为"数据类别"。

8. **四要素矩阵逐行对应与重复数据展示**：每行对应文档中的一行/一条数据，不得将多行合并。如果同一数据项出现在多个页签，仍应分别列出，标注"（与XX页签R5相似，重复出现）"——不同行的存留期限可能完全不同，合并会导致混淆。

   **重复用户组数据展示规则**：当多个用户组（如1.1华为员工/1.2外包员工/1.3运营商雇员）的数据表格内容完全相同时（数据项、用途、处理方式、存留期限逐字一致），四要素矩阵仅展示1组数据作为代表，在展示行后标注"（1.2/1.3与1.1完全相同，不重复展示）"。统计仍按实际行数计算（如3组×2类型=6项），不因展示简化而减少。**判断标准**：仅当所有四要素内容逐字一致时才可合并展示；如果存留期限或处理方式有任何差异，必须分行列出。

9. **△的轻重标注**：对处理方式和存留期限中的△项，进一步标注轻重，帮助读者区分优先级：
   - **△（重）**：敏感数据（密码、密保问题、生物特征）缺加密措施；持久化存储数据无加密；第三方共享缺加密说明；**持久化存储数据无明确存留时限（如"永久存储""长期保留""注销后删除""注销后30天删除""手动删除""条数/容量限制型"）**——注意"注销后X天删除"的X天是注销后的清理窗口而非从收集开始的存留上限，数据在注销前无限期存储，仍标△（重）；敏感数据持久化存储存留期限仅策略描述
   - **△（轻）**：端侧临时使用/仅读取数据缺加密声明；不存储数据但未明确写"不保存"；**端侧临时使用/仅读取数据存留期限未明确写"不保存"**；传输加密但未补充说明存储方式
   在四要素矩阵中用"△（重）""△（轻）"标注。最终状态判定不受轻重影响（都是△），但缺失项详细说明表中应按轻重排序，重优先。**关键区分**：只要是持久化存储的数据，其存留期限△一律标△（重）；只有不持久化场景（端侧临时/仅读取）的△才标△（轻）。

10. **判定结论严格执行**：
    - 所有要素均为✓ → PASS
    - 任何一项要素存在✗ → **FAIL**，不得因✗数量少而放宽
    - 仅存在△（无✗）→ WARNING

11. **统计口径统一**：报告中涉及两套统计，必须分两个表呈现，不得混淆：
    - **四要素状态统计表**：按数据类型的最终状态统计（每项只计入一个类别），格式为 `| ✓合规 | △模糊 | ✗不合规 | 合计 |`，三项之和=数据类型总数
    - **各要素判定分布表**：按要素维度分别统计，格式为 `| 要素 | ✓ | △ | ✗ |`，每个要素的✓+△+✗=数据类型总数
    审计结论段中提及"模糊项"数量时，必须明确是"X项数据类型存在△问题"（对应状态统计表），而非"X项要素△"（对应分布表）。两种口径数字通常不同，混用会导致读者困惑。

### 阶段四自检

完成四要素检查后、进入下一阶段前，逐项确认：

- [ ] **跨章节关联验证**（模式B/C）：统一声明章节中的每条要素是否都已关联到数据类型
- [ ] **模式识别自洽性**：识别的模式与检查策略是否一致。如果标注为模式A，检查是否存在四要素分散在其他章节的情况——如果有，则模式识别有误，应修正为模式B或C并执行跨章节关联
- [ ] **存留期限✗/△判定校验**：逐一检查每个标✗的存留期限项，问自己"文档是否在任何地方（包括统一声明章节）触及了该数据的存留问题？"如果文档承认收集了该数据且存在统一存留声明（如"卸载后删除"），则该数据类型的存留期限不应标✗，应标△（引用统一声明）。只有文档对该数据类型的存留问题完全未触及才标✗
- [ ] **端侧临时数据存留期限校验**：对处理方式已判定为"端侧临时使用""仅读取""不保存"的数据类型，检查其存留期限是否被标✗——如果是，大概率偏严。端侧临时数据如果上下文暗示不存储，存留期限应标△（轻）而非✗
- [ ] **△比例合理性**：存留期限列△占比低于20%（数据类型>20时），大概率判定过松——回查是否只检查了是否为空
- [ ] **处理方式判定合理性**：处理方式列✓占比低于20%时（数据类型>5时），大概率判定过严——回查是否对端侧临时使用/仅读取/删除类场景也机械要求了加密措施。合规隐私声明中端侧临时数据不声明加密是常见且合理的
- [ ] **非个人数据识别充分性**：3.3节是否按组分析而非一行概括？是否有至少2个边界组被讨论？是否所有"属于个人数据"的判定都有针对该组数据的具体理由（而非统一用"可间接识别用户"一句话带过）？
- [ ] **缺失项名称完整性**："数据类型"列是否都填写了实际名称，"当前描述"列是否引用了原文
- [ ] **四要素矩阵完整性**：是否列出了所有✗和△项，不得用"..."省略
- [ ] **统计数字校验**：合规项数 + 模糊项数 + 不合规项数 = 数据类型总数（等于所有页签/表格数据行数之和，不得估算）。逐行扫描状态列计数，不得凭印象估算。统计数字不得使用"约""~""+"等模糊表述，必须是精确整数
- [ ] **DI清单留存期"不被留存"与"未定义"区分**（仅数据控制者）：4D-1中是否正确区分了"不被留存的数据"（合法）和"被留存但未定义留存期的数据"（违规）
- [ ] **留存期上限比对类型匹配**（仅数据控制者）：4D-2中数据类型匹配是否基于 Read 读取的真实内容而非 LLM 推断？匹配不准时是否标记为 WARNING 而非 FAIL
- [ ] **一致性比对真实性**（仅数据控制者）：4D-3中的类型匹配和时长比对是否基于 Read 读取的真实内容？是否存在 LLM 推断或猜测的匹配关系？
- [ ] **删除能力检查范围**（仅设备供应者）：4D-4中关键词扫描无命中时是否扩大关键词范围再试一次？对二进制格式文档是否使用了 Read 工具替代 Grep？

### 阶段四D：留存期合规性深度检查（RL-11.4.1-1）

#### 数据源说明（重要）

> 阶段四检查隐私声明（数据控制者）或个人数据说明（设备供应者）中的存留期限描述完整性——判定对象是隐私声明/个人数据说明文档中的文字描述。
>
> 阶段四D检查DI清单中的留存期数值（数据控制者）或项目文档中的删除能力描述（设备供应者）——判定对象是DI清单 Excel 中的列值（数据控制者）或项目文档中的文字描述（设备供应者）。
>
> 两者数据源不同，判定互不覆盖：阶段四✓表示隐私声明中有存留期限描述，4D-1 FAIL表示DI清单中留存期字段为空。两者可以并存，不构成矛盾——前者说明"声明层面有描述"，后者说明"清单层面数据不完整"。
>
> 4D步骤的⊘不影响阶段四的内部判定结果（阶段四✓仍为✓，✗仍为✗），但⊘计入整体最终判定时产生WARNING，需人工复核文档完整性。

#### 条件执行指令

```
if 模式 == 数据控制者:
    执行 4D-1（留存期完整性校验）
    执行 4D-2（留存期上限合规校验）
    执行 4D-3（留存期对外一致性校验）
elif 模式 == 设备供应者:
    执行 4D-4（删除能力存在性校验）
```

#### 4D-1：留存期完整性校验（仅数据控制者）

**前提条件**：DI 清单已成功定位（阶段二确认结果非⊘）。若 DI 清单为⊘，4D-1直接标记⊘（无法检查），跳过本步。

**检查内容**：DI 清单中是否存在被留存但留存期字段为空或未定义的个人数据。

**判定规则**：
- 留存期标记为"不留存""不存储""不保存"等 → 合法，不计入缺失
- 留存期为空或未填写 → FAIL（被留存但未定义留存期）
- 留存期有具体值或策略描述 → PASS

#### 4D-2：留存期上限合规校验（仅数据控制者）

**前提条件**：DI 清单和留存规范均已成功定位。任一为⊘，4D-2直接标记⊘（无法检查），跳过本步。

**比对逻辑**：
1. 先进行数据类型匹配（清单条目 → 规范条目），类型匹配不准会导致误判
2. 将清单中的留存期时长与规范中对应类型的上限时长做比较
3. 留存期超过上限 → FAIL
4. 数据类型在规范中找不到明确对应 → WARNING（不标 FAIL，避免误判）
5. **非数值型留存期**：DI清单中可能存在策略性描述（如"留存至业务目的达成""注销后删除""永久存储"），无法与规范上限做数值比较 → WARNING（"留存期为策略描述'XX'，无法进行数值比对，需人工确认是否合规"）

**重要原则**：类型匹配必须基于 Read 工具读取的真实内容，禁止 LLM 推断或猜测匹配关系。如果 DI 清单中的数据类型在留存规范中找不到明确对应，标记为 WARNING 而非强行匹配。

#### 4D-3：留存期对外一致性校验（仅数据控制者）

**前提条件**：DI 清单和隐私声明均已成功定位。任一为⊘，4D-3直接标记⊘（无法检查），跳过本步。

**比对维度**：
- 数据类型是否在隐私声明中有所覆盖（遗漏检查）
- 留存期时长是否与隐私声明中的描述一致（时长匹配检查）

**比对逻辑**：
1. 复用阶段三已提取的隐私声明留存期描述
2. 将 DI 清单中每条数据的留存期与隐私声明中对应数据类型的留存期描述进行比对
3. 清单中有但隐私声明未提及 → FAIL（遗漏）
4. 留存期时长不一致 → FAIL（时长不匹配）
5. 类型匹配不确定 → WARNING
6. **隐私声明描述模糊**：隐私声明中对应数据类型的存留期限为△（如"短期保存""根据业务需要"等模糊描述），无法与DI清单中的具体值精确比对 → WARNING（"隐私声明存留期限描述为'XX'（△模糊），无法与DI清单精确比对，需人工确认"）
7. **DI清单留存期为策略描述**：DI清单中该条目的留存期为策略性描述（如"留存至业务目的达成""注销后删除""永久存储"），无法与隐私声明中的具体值精确比对 → WARNING（"DI清单留存期为策略描述'XX'，无法与隐私声明精确比对，需人工确认"）

**重要原则**：一致性比对必须基于 Read 工具读取的真实内容，禁止 LLM 推断或猜测匹配关系。

#### 4D-4：删除能力存在性校验（仅设备供应者）

**第一步——关键词扫描**：

使用 Grep 在项目文档目录全文搜索以下删除类关键词：
- 删除个人信息、删除隐私数据、删除个人数据、remove personal data、erase personal information
- 数据销毁、data erasure、数据擦除、clear personal data

**第二步——扩大搜索（如第一步无命中）**：

追加以下关键词再试一次：清除数据、erase、wipe、重置、reset、匿名化、anonymize、脱敏、desensitize

**第三步——LLM 判断**：

对命中关键词的文档，使用 Read 读取相关段落，LLM 判断是否描述了用户删除个人数据的方法或入口。

**判定规则**：
- 找到至少一个文档描述了删除个人数据的方法或入口 → ✓
- 仅找到关键词但未描述具体方法/入口 → ⚠️
- 扩大搜索后仍无命中 → ✗

**二进制格式说明**：对 xlsx/chm 等二进制格式文档，Grep 全文搜索可能无法命中关键词，此时应使用 Read 工具读取文档内容后由 LLM 判断是否包含删除能力描述。

**注意**：本步骤检查"有没有删除个人数据能力描述"，不深入验证机制的具体实现和有效性。

#### 工具策略

阶段四D与现有阶段保持一致，使用分层降级策略：

1. **MCP 可用时**：优先使用 `scan_documents`（定位文件）、`extract_tables`（提取 xlsx 表格）、`search_in_document`（关键词搜索）、`read_document`（读取内容）
2. **MCP 不可用 → Read 优先**：尝试使用 Read 工具读取 xlsx/docx/pdf 等文件（opencode 的 Read 工具支持多种格式，可获取页签名和列值）
3. **Read 无法满足结构化提取需求时 → Bash+Python**：使用 Bash+Python 脚本（openpyxl 读取 xlsx 等），与现有降级策略一致
4. 在报告中标注降级模式

#### ⊘（无法检查）标记说明

当文档缺失导致某个4D步骤无法执行时，使用⊘标记：
- ⊘ 不等同于 ✓（不表示通过）
- ⊘ 不等同于 ✗（不表示失败，因为缺乏检查依据）
- ⊘ 不等同于 △（不表示模糊，而是信息不足）
- ⊘ 表示"因文档缺失无法检查"，需人工复核
- ⊘ 计入最终判定时：产生 WARNING（需人工复核文档完整性），不产生 FAIL

### 阶段四C：非个人数据识别执行指导

3.3节是报告中最容易被敷衍的章节。以下是必须遵守的执行规范——**强制使用8组模板，逐行填写，不得跳过任何组**。

**强制8组表格模板**——3.3节必须使用以下表格格式，8个标准组逐行填写。文档中未涉及的组写"不涉及"，不得省略该行。边界组（标注★）必须回答3个问题并给出具体理由：

| 组别 | 本文涉及的数据项 | 判定 | 理由（边界组须回答3问题） | 位置 |
|------|----------------|------|------------------------|------|
| 用户身份类 | 姓名、手机号、邮箱、工号、用户账号、用户口令等 | 个人数据 | 直接标识自然人，无需讨论 | — |
| 生物特征类 | 人脸、指纹、声纹等 | 个人数据（敏感）/ 不涉及 | 直接标识自然人/文档未涉及 | — |
| 网络标识类 ★ | IP地址、MAC地址、SIP号码等 | 个人数据 / 可能非个人数据 / 不涉及 | ①能否直接识别？②能否间接识别？③是否关联用户行为？ | — |
| 设备标识类 ★ | SN、设备ID、设备名称等 | 个人数据 / 可能非个人数据 / 不涉及 | ①能否直接识别？②能否间接识别？③是否关联用户行为？ | — |
| 系统配置类 ★ | OS类型/版本、语言配置、时区等 | 个人数据 / 可能非个人数据 / 不涉及 | ①能否直接识别？②能否间接识别？③是否关联用户行为？ | — |
| 日志记录类 ★ | 运行日志、操作日志、故障日志等 | 个人数据 / 可能非个人数据 / 不涉及 | ①能否直接识别？②能否间接识别？③是否关联用户行为？ | — |
| 业务数据类 | 用户文件、照片、视频等 | 个人数据 / 不涉及 | 与用户行为/身份关联 | — |
| 运维配置类 ★ | SNMP社区名、Webhook URL、证书、AK/SK等 | 可能非个人数据 / 个人数据 / 不涉及 | ①能否直接识别？②能否间接识别？③是否关联用户行为？ | — |

**填写规则**：
1. **"本文涉及的数据项"列**：文档中出现的该组数据项，逐项列出；文档未涉及该组写"—"
2. **"判定"列**：个人数据 / 可能非个人数据 / 非个人数据 / 不涉及（四选一）
3. **"理由"列**：非边界组（用户身份类/生物特征类/业务数据类）如判定为"个人数据"，可写"直接标识自然人，无需讨论"等简短理由；边界组（★）**必须**回答3个问题（①能否直接识别自然人？②能否间接识别？③是否关联用户行为？），将答案融入理由列，每组的理由必须针对该组数据的具体特征
4. **"位置"列**：标注该组数据在文档中的位置（如Table 3 R1）；不涉及组写"—"
5. **不得跳过任何组**：即使文档明显不涉及某组（如生物特征类），也必须保留该行并写"不涉及"——这证明检查了所有标准组
6. **口令/密码的归类**：用户口令/密码归入"用户身份类"（认证凭证与用户身份绑定），不得归入"运维配置类"。运维配置类指系统运维参数（SNMP社区名、Webhook URL、证书私钥等），与用户身份无直接绑定关系

**三问题判定规则**（仅适用于★边界组）：
1. 该组数据能否直接识别特定自然人？（是→个人数据）
2. 该组数据能否与其他数据结合间接识别自然人？（是→个人数据）
3. 在本文档的业务场景中，该组数据是否与用户行为/偏好/身份关联？（是→个人数据）

三个问题均为否→判定为"可能非个人数据"或"非个人数据"。任一为是→判定为"个人数据"。

**不合格写法**（常见错误，必须避免）：
- ❌ 省略"不涉及"的组——必须8行全部填写
- ❌ 一行概括"所有项均属于个人数据，因为均涉及用户信息收集"——没有分组讨论
- ❌ 全部数据类型都标"属于个人数据"且理由都是同一句话（如"可间接识别用户"）——没有针对每组数据的具体分析
- ❌ 将口令/密码归入"运维配置类"——口令属认证凭证，归用户身份类
- ❌ 只列出判定为"非个人数据"的项，对判定为"个人数据"的项不说明理由——所有边界组都须给出理由

### 阶段四B：文档质量异常检测

检查文档自身的逻辑一致性，写入报告附录，不影响四要素判定：

1. **多用户组数据一致性**：按用户角色分组时，各组数据表格内容完全相同 → 标注异常
2. **法律基础与数据类型匹配**：不同法律基础对应相同数据处理方式 → 标注⚠️
3. **声明覆盖范围一致性**：第一节列出N种数据类型，存留期限章节只提到M种（M<N）→ 逐一确认遗漏

**文档质量异常表的格式**：必须使用表格格式，列结构为 `| # | 异常类型 | 说明 | 风险等级 |`，不得使用编号列表或自由文本替代。无异常时写"未检测到文档质量异常"而非填入正常项。

### 阶段五（仅设备供应者模式）：格式合规检查

5项格式要求，严格判定，不得因"基本满足"而标✓：

1. **分类结构**：按数据类别分组（✓）/ 按功能场景分组（⚠️）/ 平铺无分组（✗）。区分：数据类别按信息性质分组（基本信息类/位置信息类/设备信息类/日志信息类）；功能场景按产品功能分组（远程通知/WLAN资源/设备管理）——场景分组≠类别分组，因为每组内混杂了多种性质的数据
2. **边界清晰**：每个数据类型有明确边界（✓）/ 存在"等""等等"开放式结尾（⚠️，即使仅1-2处）/ 大量开放式列举（✗）
3. **数据来源标注**：每种数据类型标注来源方式（✓）/ 仅部分标注或有空值（⚠️）/ 未标注（✗）
4. **版本与修订记录**：版本号+修订记录（✓）/ 仅日期或仅版本号（⚠️）/ 无（✗）
5. **删除/匿名化能力**：文档描述了用户删除个人数据的方法或入口（✓）/ 仅找到关键词但未描述具体方法（⚠️）/ 未找到任何删除能力描述（✗）。详见阶段四D-4删除能力存在性校验

### 阶段六：生成报告

直接按报告模板生成，不需要询问用户是否需要生成。保存在被检测文档所在目录下：
- 数据控制者模式：`隐私声明合规检测报告.md`
- 设备供应者模式：`个人数据说明合规检测报告.md`

报告模板见：
- 数据控制者模式：`references/report-template-mode1.md`
- 设备供应者模式：`references/report-template-mode2.md`

**报告章节结构**：

**数据控制者模式**——在"三、四要素完整性检查"之后插入新章节：
- 三、四要素完整性检查（现有）
- 四、留存期合规性深度检查（RL-11.4.1-1）（新增，含4.0数据源定位 + 4.1/4.2/4.3）
- 五、第三方SDK/共享信息检查（原为"四"，编号顺延）
- 六、附录（原为"五"，编号顺延，含审计范围/审计说明/文档质量异常子节）

**设备供应者模式**——在"三、四要素完整性检查"之后插入新章节：
- 三、四要素完整性检查（现有）
- 四、留存期合规性深度检查（RL-11.4.1-1）（新增，含4.0数据源定位 + 4.4删除能力校验）
- 五、格式合规性检查（原为"四"，编号顺延）
- 六、附录（原为"五"，编号顺延，含审计范围/审计说明/文档质量异常子节）

**子节模式归属**：4.1-4.3仅在数据控制者模式下生成，4.4仅在设备供应者模式下生成。非适用模式的子节直接省略，不写"不适用"。4.0数据源定位表根据模式只列出相关数据源行。

**报告判定逻辑**（与原11.1.1-1+2的△→WARNING规则兼容）：

| 模式 | PASS | FAIL | WARNING |
|------|------|--------|---------|
| 数据控制者 | 阶段四全部✓ + 4D无FAIL + 无⊘ + 4D无WARNING | 阶段四有✗ 或 4D-1/2/3任意一步有FAIL | 阶段四有△（无✗）或 4D无FAIL但存在⊘ 或 4D-2/4D-3存在WARNING |
| 设备供应者 | 阶段四全部✓ + 格式合规全部✓ + 4D-4为✓ | 阶段四有✗ 或 格式合规有✗ 或 4D-4为✗ | 阶段四有△（无✗）或 格式合规有⚠️ 或 4D-4为⚠️/⊘ |

**判定优先级**：FAIL > WARNING > PASS。任一条件命中更高优先级即取更高优先级。

**关键判定强调**（反复出现误判的场景）：4D-2留存期上限合规校验中，如果存在超上限条目（留存期超过规范上限），这些条目标FAIL而非WARNING——FAIL是留存期数值超过上限的明确违规，WARNING仅用于类型匹配不确定或非数值型留存期无法比对的情况。因此，4D-2有任何超上限FAIL条目 → 整体结论必须标FAIL，不得标WARNING。同理，4D-1有被留存但未定义留存期的条目（FAIL）、4D-3有遗漏或不一致条目（FAIL）→ 整体结论必须标FAIL。4D步骤中仅WARNING和⊘不触发整体FAIL。

**⊘与阶段四的关系**：4D步骤的⊘不影响阶段四的内部判定结果（阶段四✓仍为✓，✗仍为✗），但⊘计入整体最终判定时产生WARNING，需人工复核文档完整性。

**报告章节结构强制对齐**：报告的章节标题、编号、顺序必须与模板一字不差，不得同义替换（如用"基本信息"替代"审计概述"）、不得重编号（如将"四"改为"5"）、不得跳号（如从"阶段四B"直接跳到"阶段六"）。模板中的每个章节都必须出现在报告中，即使该章节内容为"无"或"不适用"，也必须保留章节标题并说明原因——不得直接省略章节。

**报告完整性校验**：生成报告后，必须对照模板逐章节确认结构完整性。特别是以下章节常被遗漏，必须确认存在：
 - 空值检测（2.2节）——即使全部无空值也必须写"空单元格数：0"，docx格式写"空单元格数：0（非表格文档，逐要素检查无空白要素项）"，markdown表格写"空单元格数：0（markdown表格文档，逐行逐列检查无空白单元格）"——不得将markdown表格文档标为"非表格文档"
- 非个人数据识别（3.3节）——不得一刀切声明全部为个人数据，必须按组逐项分析，**必须覆盖8个标准组（含"不涉及"组）**
- 格式合规性检查（第五章，仅设备供应者模式）
- 缺失项详细说明（3.2节）——必须区分✗项和△项两个子表
- 降级模式标注（附录审计说明中）——无论是否使用MCP工具都须标注
- △轻重标注（3.2.2节）——△模糊项按重轻排序
- 文档质量异常表——无异常时写"未检测到文档质量异常"而非填入正常项

**自动校验脚本**：报告生成后，应运行 `references/validate_report.py <报告路径> [--mode 1|2]` 进行自动校验。脚本检查项包括：章节结构完整性、审计概述4字段、统计数字加和、3.3节8组全覆盖、降级模式标注、△轻重标注、结论与统计一致性。校验FAIL时必须修复报告中的问题后重新校验，直到PASS。

生成后告知用户完整路径。

## 注意事项

1. **两种模式严格互斥**：每次只执行一种，不得中途切换
2. **3轮搜索后直接结论**：不询问用户确认缺失，报告中列出搜索范围供核实
3. **表格优先**：提取数据时优先用extract_tables（MCP）或Python脚本（降级）
4. **保守判定**：无法确定是否合规时标记"待确认"而非强行归类
5. **脱敏展示**：报告中不得明文显示口令等敏感数据，用****替代
6. **不修改文档**：此skill仅检查和报告，不修改任何被检查的文档
7. **MCP不可用不得中断**：自动降级，**所有报告必须在附录审计说明中标注降级模式**（如"本次检测使用降级模式（Python脚本读取docx文件）"），无论是否使用了MCP工具
8. **保持运行**：整个检测流程从参数收集到报告生成必须连续完成
9. **四要素矩阵行数控制**：≤30全列，>30列所有✗△项+5-10个✓代表项，其余省略
10. **"收集方式"≠"处理方式"**：很多docx隐私声明表格有"收集方式"列（内容如"APP收集""用户输入"），这只是描述数据如何被获取，不是处理方式。处理方式关注的是数据获取之后如何存储、加密、共享。不得将"APP收集"作为处理方式的描述——这是docx格式下最常见的混淆
11. **文档质量异常表"无异常"写法**：如果检测后未发现异常，不得将"覆盖一致"等非异常内容填入异常表——应在表中写"未检测到文档质量异常"，而非填入"正常"项
12. **DI清单多文件处理**：DI清单扫描到多个时，4D-1/2/3标记⊘（"存在多个DI清单文件，无法确定使用哪个"），标 WARNING 而非强行选择
13. **留存规范多文件处理**：留存规范扫描到多个时，4D-2标记⊘（"存在多个留存规范文件，无法确定使用哪个"），标 WARNING
14. **4D步骤工具策略**：优先使用MCP工具；降级模式下使用Glob/Grep/Read + Bash+Python脚本（与现有降级策略一致），在报告中标注降级模式
15. **4D步骤实事求是**：类型匹配和一致性比对必须基于 Read 工具读取的真实内容，禁止 LLM 推断或猜测匹配关系。禁止编造留存期定义、处置机制或删除能力描述。如果 DI 清单中的数据类型在留存规范或隐私声明中找不到明确对应，标记为 WARNING 而非强行匹配。如果文档中无相关内容，应明确标注⊘或"文档中未发现相关描述"，不得凭空推断
16. **⊘标记语义**：⊘表示"因文档缺失无法检查"，不等同于✓（通过）、✗（失败）或△（模糊）。⊘计入最终判定时产生 WARNING，需人工复核文档完整性
17. **边缘场景判定指导**：以下场景在实际检测中频繁出错，判定时须特别注意：
    - **存留期限边界值**：留存规范上限为"180天"，DI清单写"180天"→✓（恰好等于上限合规）；写"181天"→4D-2 FAIL（超上限1天）；写"默认180天，可配置1-365天"→△（可配置超过上限）
    - **混合格式文档**：同一文档中部分章节用表格、部分章节用叙述式文本→识别为模式C，不得只提取表格部分而忽略叙述章节中的要素
    - **空表格/空页签**：xlsx文档存在空页签（0行数据）→排除不计入总数据行数，但须在数据来源结构表中列出并标注"0行数据，排除"
    - **多页签部分重叠**：不同页签包含相同数据类型但留存期不同→分别列出（存留期限可能不同），不得合并
    - **"注销后X天删除"vs"保存X天"**："注销后30天删除"→△（重）（注销前无限期存储）；"保存30天"→✓（明确从收集起30天）；"自收集起保存30天，注销后30天内清理"→✓（既有上限又有清理窗口）
    - **跨报告一致性**：同一数据在不同模式报告中应使用相同判定标准——如"注销后30天删除"在Mode1标△(重)时，Mode2中对同一描述也应标△(重)，不得一个标✓一个标△
