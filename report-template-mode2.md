# 模式二报告模板（设备供应者）

阶段七生成报告时，按此模板输出。

```markdown
# 个人数据说明合规检测报告

## 审计概述

| 项目 | 内容 |
|------|------|
| 审计条目 | 红线条目 11.1.1-2 个人数据说明满足公司要求 |
| 检查模式 | 设备供应者 |
| 审计结论 | **PASS** / **FAIL** / **WARNING** |
| 审计日期 | {YYYY-MM-DD} |

## 审计结论

{2-3句话总结}

## 一、文档发现

| # | 文件名 | 格式 | 匹配评分 | 数据来源 |
|---|--------|------|---------|---------|

{文档缺失时同模式一格式}

## 二、内容提取

### 2.1 数据来源结构
| 文件 | 提取方式 | 提取内容 |
|------|---------|---------|

### 2.2 空值检测（不得跳过此章节）

**空值总览**

| 文件 | 数据区域 | 总数据行 | 空单元格数 |
|------|---------|---------|-----------|

⚠️此章节不得省略或跳过。即使全部列无空值，也必须明确写出"空单元格数：0，各列均无空值"。对于docx等非表格格式文档，不得写"不适用表格结构"，应写"空单元格数：0（非表格文档，逐要素检查无空白要素项）"。

**空值明细**（按数据区域逐列列出，仅列出存在空值的列；全部列无空值时写"各列均无空值"）

| 数据区域 | 列名 | 空值数 | 空值所在行 |
|---------|------|-------|-----------|
| {页签名/sheet名} | {列名} | {n} | R10, R11, R12 |

**空值检测要求**：必须给出精确数字，不得使用"约""~""<5%""少数""个别""无明显空值""~100行"等任何估算或模糊表述。总数据行数必须是所有页签数据行数的精确求和。空单元格数必须是一个精确整数，不得是"约5个""个别"等。

## 三、四要素完整性检查

### 3.1 四要素矩阵

**符号说明**：✓ = 合格（有明确、具体的描述）| △ = 模糊（有描述但不具体或不完整）| ✗ = 缺失（无相关描述）

| # | 使用场景 | 收集的个人数据项 | 类型描述 | 目的 | 处理方式 | 存留期限 | 状态 |
|---|---------|-----------------|---------|------|---------|---------|------|

**四要素矩阵要求**：
- 矩阵必须包含"收集的个人数据项"列，展示原文该列的关键内容（如"姓名、电话号码、邮箱"），不得只有场景名。数据项过多时可截取前3-5项+加"...等N项"
- "使用场景"列填原文场景名；"收集的个人数据项"列填原文对应的数据项名称
- 当数据类型总数 ≤ 30 时，必须逐一列出所有行
- 当数据类型总数 > 30 时，可以列出关键代表行（包含所有✗和△项，以及5-10个✓代表项），其余用"..."省略
- 不得省略任何✗或△项——这些是报告的核心发现
- 统计数字必须精确，使用状态统计表展示

**四要素状态统计**：

| 状态 | 含义 | 数量 |
|------|------|------|
| ✓ 合规 | 全部4项要素均为✓ | {n}项 |
| △ 模糊 | 存在△但无✗ | {n}项 |
| ✗ 不合规 | 存在至少1项✗ | {n}项 |
| **合计** | | **{N}项** |

**统计口径**：按数据类型的最终状态统计，每个数据类型只计入一个类别，不按要素拆计：
- 如某数据类型有任一要素为✗ → 整项计入"不合规"（即使同一类型还有△要素，也不额外计数）
- 如某数据类型有△但无✗ → 整项计入"模糊"（即使有多个△要素，也只计1次）
- 如某数据类型所有要素均为✓ → 整项计入"合规"

三项数字之和必须等于数据类型总数。如果不等，说明统计有误，需回查。
常见错误：某数据类型有2个△要素，统计时被计为2项模糊——这是错的，应只计1次。

### 3.2 缺失项详细说明
| 数据类型 | 缺失要素 | 当前值 | 合规要求 | 来源位置 |
|---------|---------|--------|---------|---------|

**数据类型列要求**：必须填写该行实际的数据类型名称（从文档A列或对应字段读取），不得只写行号（如"(R163)"）——行号只是定位信息，放在来源位置列。
**当前描述列要求**：不得统一写"未说明""未提及""描述转储策略但未明确时限""未说明具体时限"。必须区分两种情况：
- 原文确实没有任何相关描述 → 写"（原文无相关描述）"
- 原文有相关描述但不满足合规要求 → 引用原文实际内容（可截取关键短语并标注行号/页签）
  例如：不应写"未说明"，应写"原文写'因功能需要，此数据默认长期保留'(eSight表R19)，无具体天数"
  例如：不应写"描述转储策略但未明确时限"，应写"原文写'达到审计日志转储条件后转储到硬盘，达到审计日志硬盘删除条件后永久删除'(eSight表R7)，仅有条件删除策略无具体时限"
  这样读者能区分"真的没写"和"写了但不合规"，判断依据完全不同。
**来源位置要求**：必须给出精确位置（如"R45"表示Excel第45行、"第1.2节(3)"表示文档第1.2节第3项、"L120-L125"表示Markdown第120-125行），不得使用"约""~""大概"等模糊定位。

### 3.3 非个人数据识别（不得跳过此章节）

**禁止一刀切声明**：不得将全部数据类型统一判为"个人数据"并给一个笼统理由（如"均涉及用户/设备信息收集"）。这种声明没有分析价值，等于跳过了此章节。**典型错误示例**：分4组（用户身份/设备信息/日志/资产信息）全部判"个人数据"，每组只写一句理由——这不叫"逐组讨论"，叫分组后一刀切。正确的做法是对边界组内的每个具体数据类型单独讨论。

对文档中列出但可能不属于个人数据的数据类型，在此章节**逐项**标注并给出判定理由。边界数据必须讨论而非忽略：

| 数据类型 | 判定 | 理由 | 位置 |
|---------|------|------|------|

**逐项识别方式**：
1. 将文档中所有数据类型按性质分组（用户身份类/网络标识类/系统配置类/日志记录类/业务数据类/运维配置类等）
2. 对每组给出判定：整组属于个人数据/非个人数据/待确认
3. 对边界组内的具体数据类型，逐一讨论为什么该类型属于或不属于个人数据

**常见边界数据类型**（必须在此章节逐一讨论，不得跳过）：
- IP地址/MAC地址：GDPR认定为个人数据，但单纯设备MAC存在争议；如果关联到自然人则属于个人数据
- 系统配置参数（如SNMP社区名）：如果不关联自然人则可能不属于个人数据
- 设备指纹/设备ID：如果可用于追踪自然人则属于个人数据
- 日志中的技术数据：如果包含用户操作记录则属于个人数据
- 安全凭证（如证书、密钥、应用Token/Secret）：本身不属于个人数据，但如果关联到特定自然人则属于
- Webhook URL/运维配置：属于系统配置，不属于个人数据
- 资产编号/物理位置/所属资源池：设备属性，不属于个人数据
- 搜索记录/搜索关键词：是否关联到特定用户，需具体分析

**最低要求**：即使所有数据类型最终都判定为个人数据，也必须按组逐一说明为什么该组属于个人数据，而非一句话概括。

## 四、格式合规性检查

| 检查项 | 结果 | 说明 |
|--------|------|------|
| 数据分类结构 | | 按数据类别分组（✓，如基本信息类/位置信息类/设备信息类/日志信息类）/ 按功能场景分组（⚠️，如远程通知/WLAN资源/设备管理——场景≠类别。⚠️注意：很多文档按"使用场景"组织（如创建账户/远程通知/IP管理），每组内混杂了多种性质的数据，这属于功能场景分类，不是数据类别分类）/ 无分组（✗） |
| 数据类型边界定义 | | 列出含"等"的具体位置和数量 |
| 数据来源标注 | | 标注覆盖率（几行有来源/总行数），列出空值所在行 |
| 版本信息 | | 版本号+修订记录（✓）/ 仅日期或仅版本号（⚠️）/ 无（✗） |
```
