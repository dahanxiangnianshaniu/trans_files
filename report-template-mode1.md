## 三、个人数据类型四要素检查

### 3.1 四要素矩阵

**符号说明**：✓ = 合格（有明确、具体的描述）| △ = 模糊（有描述但不具体或不完整，轻/重标注）| ✗ = 缺失（无相关描述）

| # | 数据类型 | 收集的个人数据项 | 类型描述 | 目的 | 处理方式 | 存留期限 | 状态 |
|---|---------|----------------|---------|------|---------|---------|------|

**四要素状态统计**：

| 状态 | 含义 | 数量 |
|------|------|------|
| ✓ 合规 | 全部4项要素均为✓ | {n}项 |
| △ 模糊 | 存在△但无✗ | {n}项 |
| ✗ 不合规 | 存在至少1项✗ | {n}项 |
| **合计** | | **{N}项** |

**统计口径**：按数据类型的最终状态统计，每项只计入一个类别——任一要素为✗→不合规；有△无✗→模糊；全✓→合规。三项之和必须等于数据类型总数。

**各要素判定分布**：

| 要素 | ✓ | △ | ✗ |
|------|---|---|---|
| 类型描述 | {n} | {n} | {n} |
| 目的 | {n} | {n} | {n} |
| 处理方式 | {n} | {n} | {n} |
| 存留期限 | {n} | {n} | {n} |

**分布口径**：按要素维度统计，每个要素的✓+△+✗=数据类型总数。此表与状态统计表口径不同——状态统计表按数据类型整体状态计（每项只计一次），本表按要素拆计（同一数据类型可出现在多个要素的△列）。

**四要素矩阵必须包含"收集的个人数据项"列**：矩阵不能只有数据类型名称，必须展示原文描述的具体数据项（如"人脸照片""WiFi名称、蓝牙名称"），否则读者无法判断类型描述是否合理。

**重复数据类型处理**：如果文档不同章节描述的是同一种数据（如1.1.2节"获取设备位置"和1.1.3节"位置信息"可能指向同一数据），应合并为一条，标注"（原文在1.1.2和1.1.3均有提及，可能为同一数据类型）"。

### 3.2 缺失项详细说明

#### 3.2.1 ✗缺失项
| 数据类型 | 缺失要素 | 当前描述 | 合规要求 | 所在位置 |
|---------|---------|---------|---------|---------|

#### 3.2.2 △模糊项
| 数据类型 | 模糊要素 | 当前描述 | 合规要求 | 所在位置 |
|---------|---------|---------|---------|---------|

**数据类型列**：必须填写实际名称，不得只写行号。
**当前描述列**：区分"原文无相关描述"和"原文有描述但不合规"——后者必须引用原文实际内容（如"原文仅写'达成本声明所述目的所需期限'(第五节)，无具体时限"）。
**来源位置列**：必须给出精确位置（R45=Excel第45行、L120-L125=Markdown第120-125行、第1.2节(3)=文档第1.2节第3项）。
**△模糊项排序**：按轻重排序——△（重）优先列出（敏感数据缺加密、持久化存储无时限），△（轻）在后（端侧临时数据缺加密声明、未明确写"不保存"）。

### 3.3 非个人数据识别（不得跳过此章节）

**禁止一刀切声明**：不得将全部数据类型统一判为"个人数据"并给一个笼统理由（如"均涉及用户/设备信息收集"）。必须逐组分析边界数据类型。

对文档中列出但可能不属于个人数据的数据类型，在此章节逐项标注并给出判定理由。边界数据必须讨论而非忽略：

| 组别 | 本文涉及的数据项 | 判定 | 理由（边界组须回答3问题） | 位置 |
|------|----------------|------|------------------------|------|

**逐项识别方式**：
1. 将文档中所有数据类型按性质分为8个标准组（用户身份类/生物特征类/网络标识类/设备标识类/系统配置类/日志记录类/业务数据类/运维配置类），逐行填写
2. 对每组给出判定：个人数据 / 可能非个人数据 / 非个人数据 / 不涉及
3. 对边界组（网络标识类★/设备标识类★/系统配置类★/日志记录类★/运维配置类★）**必须**回答3个问题（①能否直接识别自然人？②能否间接识别？③是否关联用户行为？），将答案融入理由列
4. "本文涉及的数据项"列：填写文档中出现的该组数据项，逐项列出；文档未涉及该组写"—"
5. "位置"列：标注该组数据在文档中的位置（如Table 3 R1）；不涉及组写"—"

**常见边界数据类型**（必须在此章节逐一讨论，不得跳过）：
- IP地址/MAC地址：GDPR认定为个人数据，但单纯设备MAC存在争议；如果关联到自然人则属于个人数据
- 系统配置参数（如SNMP社区名）：如果不关联自然人则可能不属于个人数据
- 设备指纹/设备ID：如果可用于追踪自然人则属于个人数据
- 日志中的技术数据：如果包含用户操作记录则属于个人数据
- 安全凭证（如证书、密钥、应用Token/Secret）：本身不属于个人数据，但如果关联到特定自然人则属于
- Webhook URL/运维配置：属于系统配置，不属于个人数据
- 资产编号/物理位置/所属资源池：设备属性，不属于个人数据
- 硬件型号/操作系统类型/网卡类型：品类信息，单独不关联自然人，但与设备标识组合可能形成设备指纹

**最低要求**：即使所有数据类型最终都判定为个人数据，也必须按组逐一说明为什么该组属于个人数据，而非一句话概括。每组的理由必须针对该组数据的特征（如"IP地址可间接定位用户地理位置"），不得用统一的笼统理由（如"可间接识别用户"）带过所有组。
