1. **输入参数与 11.1.1-1+2 完全一致**：不增加任何额外用户输入项，11.4.1 原有的5个可选关键词参数全部内嵌到 skill 逻辑中

4. **数据源显式区分**：阶段四检查隐私声明/个人数据说明，阶段四D检查DI清单，两者数据源不同，判定互不覆盖

## 八、新增阶段四D：留存期合规性深度检查

### 数据源说明（重要）

> 阶段四检查隐私声明（数据控制者）或个人数据说明（设备供应者）中的存留期限描述完整性——判定对象是隐私声明/个人数据说明文档中的文字描述。
>
> 阶段四D检查DI清单中的留存期数值——判定对象是DI清单 Excel 中的列值。
>
> 两者数据源不同，判定互不覆盖：阶段四✓表示隐私声明中有存留期限描述，4D-1 FAILED表示DI清单中留存期字段为空。两者可以并存，不构成矛盾——前者说明"声明层面有描述"，后者说明"清单层面数据不完整"。
>
> 4D步骤的⊘不影响阶段四的内部判定结果（阶段四✓仍为✓，✗仍为✗），但⊘计入整体最终判定时产生WARNING，需人工复核文档完整性。

### 条件执行指令

```
if 模式 == 数据控制者:
    执行 4D-1（留存期完整性校验）
    执行 4D-2（留存期上限合规校验）
    执行 4D-3（留存期对外一致性校验）
elif 模式 == 设备供应者:
    执行 4D-4（删除能力存在性校验）
```

### 4D-1：留存期完整性校验（仅数据控制者）

**前提条件**：DI 清单已成功定位（阶段二确认结果非⊘）。若 DI 清单为⊘，4D-1直接标记⊘（无法检查），跳过本步。

**检查内容**：DI 清单中是否存在被留存但留存期字段为空或未定义的个人数据。

**判定规则**：
- 留存期标记为"不留存""不存储""不保存"等 → 合法，不计入缺失
- 留存期为空或未填写 → FAILED（被留存但未定义留存期）
- 留存期有具体值或策略描述 → PASS

### 4D-2：留存期上限合规校验（仅数据控制者）

**前提条件**：DI 清单和留存规范均已成功定位。任一为⊘，4D-2直接标记⊘（无法检查），跳过本步。

**比对逻辑**：
1. 先进行数据类型匹配（清单条目 → 规范条目），类型匹配不准会导致误判
2. 将清单中的留存期时长与规范中对应类型的上限时长做比较
3. 留存期超过上限 → FAILED
4. 数据类型在规范中找不到明确对应 → WARNING（不标 FAILED，避免误判）
5. **非数值型留存期**：DI清单中可能存在策略性描述（如"留存至业务目的达成""注销后删除""永久存储"），无法与规范上限做数值比较 → WARNING（"留存期为策略描述'XX'，无法进行数值比对，需人工确认是否合规"）

**重要原则**：类型匹配必须基于 Read 工具读取的真实内容，禁止 LLM 推断或猜测匹配关系。如果 DI 清单中的数据类型在留存规范中找不到明确对应，标记为 WARNING 而非强行匹配。

### 4D-3：留存期对外一致性校验（仅数据控制者）

**前提条件**：DI 清单和隐私声明均已成功定位。任一为⊘，4D-3直接标记⊘（无法检查），跳过本步。

**比对维度**：
- 数据类型是否在隐私声明中有所覆盖（遗漏检查）
- 留存期时长是否与隐私声明中的描述一致（时长匹配检查）

**比对逻辑**：
1. 复用阶段三已提取的隐私声明留存期描述
2. 将 DI 清单中每条数据的留存期与隐私声明中对应数据类型的留存期描述进行比对
3. 清单中有但隐私声明未提及 → FAILED（遗漏）
4. 留存期时长不一致 → FAILED（时长不匹配）
5. 类型匹配不确定 → WARNING
6. **隐私声明描述模糊**：隐私声明中对应数据类型的存留期限为△（如"短期保存""根据业务需要"等模糊描述），无法与DI清单中的具体值精确比对 → WARNING（"隐私声明存留期限描述为'XX'（△模糊），无法与DI清单精确比对，需人工确认"）
7. **DI清单留存期为策略描述**：DI清单中该条目的留存期为策略性描述（如"留存至业务目的达成""注销后删除""永久存储"），无法与隐私声明中的具体值精确比对 → WARNING（"DI清单留存期为策略描述'XX'，无法与隐私声明精确比对，需人工确认"）

**重要原则**：一致性比对必须基于 Read 工具读取的真实内容，禁止 LLM 推断或猜测匹配关系。

### 4D-4：删除能力存在性校验（仅设备供应者）

**第一步——关键词扫描**：

使用 Grep 在项目文档目录全文搜索以下删除类关键词：
- 删除个人信息、删除隐私数据、删除个人数据、remove personal data、erase personal information
- 数据销毁、data erasure、数据擦除、clear personal data

**第二步——扩大搜索（如第一步无命中）**：

追加以下关键词再试一次：清除数据、erase、wipe、重置、reset、匿名化、anonymize、脱敏、desensitize

**第三步——LLM 判断**：

对命中关键词的文档，使用 Read 读取相关段落，LLM 判断是否描述了用户删除个人数据的方法或入口。

**判定规则**：
- 找到至少一个文档描述了删除个人数据的方法或入口 → ✓
- 仅找到关键词但未描述具体方法/入口 → ⚠️
- 扩大搜索后仍无命中 → ✗

**二进制格式说明**：对 xlsx/chm 等二进制格式文档，Grep 全文搜索可能无法命中关键词，此时应使用 Read 工具读取文档内容后由 LLM 判断是否包含删除能力描述。

**注意**：本步骤检查"有没有删除个人数据能力描述"，不深入验证机制的具体实现和有效性。

### 工具策略

阶段四D与现有阶段保持一致，使用分层降级策略：

1. **MCP 可用时**：优先使用 `scan_documents`（定位文件）、`extract_tables`（提取 xlsx 表格）、`search_in_document`（关键词搜索）、`read_document`（读取内容）
2. **MCP 不可用 → Read 优先**：尝试使用 Read 工具读取 xlsx/docx/pdf 等文件（opencode 的 Read 工具支持多种格式，可获取页签名和列值）
3. **Read 无法满足结构化提取需求时 → Bash+Python**：使用 Bash+Python 脚本（openpyxl 读取 xlsx 等），与现有降级策略一致
4. 在报告中标注降级模式

### ⊘（无法检查）标记说明

当文档缺失导致某个4D步骤无法执行时，使用⊘标记：
- ⊘ 不等同于 ✓（不表示通过）
- ⊘ 不等同于 ✗（不表示失败，因为缺乏检查依据）
- ⊘ 不等同于 △（不表示模糊，而是信息不足）
- ⊘ 表示"因文档缺失无法检查"，需人工复核
- ⊘ 计入最终判定时：产生 WARNING（需人工复核文档完整性），不产生 FAILED

### 报告判定逻辑

**完整判定逻辑**（必须与原11.1.1-1+2的△→WARNING规则兼容）：

| 模式 | PASS | FAILED | WARNING |
|------|------|--------|---------|
| 数据控制者 | 阶段四全部✓ + 4D无FAILED + 无⊘ + 4D无WARNING | 阶段四有✗ **或** 4D-1/2/3任意一步有FAILED | 阶段四有△（无✗）**或** 4D无FAILED但存在⊘ **或** 4D-2/4D-3存在WARNING |
| 设备供应者 | 阶段四全部✓ + 格式合规全部✓ + 4D-4为✓ | 阶段四有✗ **或** 格式合规有✗ **或** 4D-4为✗ | 阶段四有△（无✗）**或** 格式合规有⚠️ **或** 4D-4为⚠️/⊘ |

**判定优先级**：FAILED > WARNING > PASS。任一条件命中更高优先级即取更高优先级。

**⊘与阶段四的关系**：4D步骤的⊘不影响阶段四的内部判定结果（阶段四✓仍为✓，✗仍为✗），但⊘计入整体最终判定时产生WARNING，需人工复核文档完整性。
