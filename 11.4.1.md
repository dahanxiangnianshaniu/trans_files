# RL-11_4_1-1: 个人数据留存期合规性检查技能

## 规则编号

RL-11_4_1-1

## 规则描述

数据控制者须基于业务处理的具体目的定义个人数据的留存期，对于业务目的已经达成或者留存期到期的个人数据及时处置（删除或匿名化）；设备供应者须提供个人数据处置（删除或匿名化）能力，以帮助客户满足留存期最小化原则。

## 分支选择

根据角色分流执行：

| 角色 | 执行步骤 | 核心关注点 |
|------|----------|-----------|
| 数据控制者 | 第一步至第三步 | 留存期定义完整性、上限合规性、对外一致性 |
| 设备供应者 | 第四步 | 是否提供删除个人数据的能力 |

## 输入规格

### 必需输入

- **project_docs_dir**: 项目文档目录路径，包含个人数据清单、留存规范、隐私声明等合规文档
  - 示例: `"D:\project\docs"` 或 `"/home/user/project/docs"`
- **role_type**: 角色类型，决定执行分支
  - 可选值: `"数据控制者"` 或 `"设备供应者"`
  - 示例: `"数据控制者"`

### 可选输入

- **personal_data_keywords**: 个人数据清单文件名关键词，默认 `["个人数据清单", "个人数据列表", "DI清单", "personal data list"]`
- **retention_period_keywords**: 留存期列名关键词，默认 `["留存期", "存储时长", "retention"]`
- **privacy_keywords**: 隐私声明文件名关键词，默认 `["隐私声明", "隐私政策", "privacy statement", "privacy policy"]`
- **standard_keywords**: 留存规范文件名关键词，默认 `["个人数据留存规范", "留存规范", "retention standard"]`
- **delete_keywords**: 删除能力关键词，默认 `["删除个人信息", "删除隐私数据", "删除个人数据", "remove personal data", "erase personal information", "数据销毁", "data erasure", "数据擦除", "clear personal data"]`

## 工具依赖

本技能通过opencode内置工具执行文件搜索和内容读取，不使用bash或MCP工具。

### 文件搜索工具

| 工具 | 用途 | 操作说明 |
|------|------|----------|
| **Glob** | 按文件名模式搜索文件 | 搜索个人数据清单、留存规范、隐私声明等文档文件 |
| **Grep** | 按关键词搜索文件内容 | 全文关键词扫描（到期处置、删除/匿名化能力等） |
| **Read** | 读取文件内容 | 读取Excel、PDF、Word等文档内容供LLM分析 |

### 关键词扫描词表

#### 第四步删除能力关键词

- 删除个人信息、删除隐私数据、删除个人数据、remove personal data、erase personal information
- 数据销毁、data erasure、数据擦除、clear personal data

## 执行流程

### 步骤0: 输入验证与环境准备

#### 0.1 验证输入参数

确认用户提供的 `project_docs_dir` 路径存在且可访问，使用 Read 工具读取目录确认内容非空。

确认 `role_type` 为 `"数据控制者"` 或 `"设备供应者"`，据此决定后续执行步骤。

#### 0.2 扫描项目文档目录

使用 Glob 工具扫描文档目录，识别以下关键文档类型：

- 个人数据清单 Excel 文件
- 留存规范 Excel 文件
- 隐私声明文档文件（PDF/Word/HTML等）
- 其他项目文档

#### 0.3 判断是否继续执行

**判断条件**：

1. `project_docs_dir` 路径有效且目录非空
2. `role_type` 参数合法
3. 关键文档文件可识别（至少能定位个人数据清单或相关文档）

**如果满足条件，继续执行后续步骤。否则，输出提示信息并结束。**

**输出格式**：

```
环境准备结果:
- 文档目录: [路径]
- 角色类型: [数据控制者/设备供应者]
- 执行步骤: [步骤1-3 / 步骤4]
- 已识别文档:
  - 个人数据清单: [文件路径 / 未找到]
  - 留存规范: [文件路径 / 未找到]
  - 隐私声明: [文件路径 / 未找到]
  - 其他文档数: [N]

开始执行个人数据留存期合规性检查...
```

---

### 第一步：完整性校验 —— 个人数据清单中每项数据是否都有留存期定义

#### 1.1 定位个人数据清单文件

使用 Glob 工具在项目文档目录下扫描文件名包含以下关键词的 Excel 文件：

- `个人数据清单`、`个人数据列表`、`DI清单`、`personal data list`

扫描模式: `**/*个人数据清单*.xlsx`、`**/*个人数据列表*.xlsx`、`**/*DI清单*.xlsx`、`**/*personal*data*list*.xlsx`

**多文件处理规则**：
- 扫描到 0 个 → 判定失败，记录失败原因
- 扫描到 1 个 → 直接使用
- 扫描到多个 → 判定失败，无法确定使用哪个清单

#### 1.2 识别有效页签

使用 Read 工具读取清单所有页签名，由 LLM 判断哪些页签包含个人数据具体内容。

**排除规则**：排除目录页、说明页、变更记录页等非数据页签。

#### 1.3 提取留存期

读取清单中留存期所在列（通常为 Q 列），列名包含关键词：`留存期`、`存储时长`、`retention`。

对有效页签中的每条个人数据，提取其留存期值。

#### 1.4 完整性判定

**失败条件**：存在被留存的个人数据但留存期字段为空或未定义。

**注意**：需区分"未被留存的数据"（留存期标记为"不留存"/"不存储"等，此为合法状态）和"被留存但未定义留存期的数据"（此为违规）。只有后者属于违规。

---

### 第二步：上限合规校验 —— 清单中的留存期是否不超过留存规范上限

#### 2.1 加载留存规范

使用 Glob 工具在项目文档目录下搜索文件名包含 `个人数据留存规范`、`留存规范`、`retention standard` 等关键词的 Excel 文件。

使用 Read 工具读取留存规范文件，按页签转为结构化数据，提取每种数据类型的留存期上限时长。

#### 2.2 逐条比对上限

将清单中每条数据的留存期与留存规范中对应数据类型的上限时长进行匹配比对。

**比对逻辑**：
1. 先进行数据类型匹配（清单条目 → 规范条目），类型匹配不准会导致误判
2. 将清单中的留存期时长与规范中对应类型的上限时长做数值比较
3. 留存期超过上限 → FAILED

#### 2.3 判定

| 条件 | 结果 |
|------|------|
| 清单中某类数据的留存期超过留存规范中定义的上限时长 | FAILED |
| 留存期未超上限 | PASS |

---

### 第三步：对外一致性校验 —— 清单留存期是否与隐私声明一致

#### 3.1 定位隐私声明文件

使用 Glob 工具在项目文档目录下扫描文件名包含以下关键词的文档：

- `隐私声明`、`隐私政策`、`privacy statement`、`privacy policy`

支持的文件格式：PDF、Word（.docx）、HTML 等。

**多文件处理规则**：
- 扫描到 0 个 → 判定失败，记录失败原因
- 扫描到 1 个 → 直接使用
- 扫描到多个 → 判定失败，无法确定使用哪个隐私声明

#### 3.2 提取留存期描述

使用 Read 工具读取隐私声明全文，将内容交给 LLM，提取其中所有关于个人数据留存期的描述。

#### 3.3 逐条比对

将清单中每条数据的留存期与隐私声明中对应数据类型的留存期描述进行比对。

**比对维度**：
- 数据类型是否在隐私声明中有所覆盖（遗漏检查）
- 留存期时长是否与隐私声明中的描述一致（时长匹配检查）

#### 3.4 判定

**失败条件**：清单中的留存期与隐私声明中的描述不一致（遗漏或时长不匹配）。

---

### 第四步：删除能力存在性校验（仅设备供应者） —— 产品是否提供删除个人数据的能力

#### 4.1 关键词扫描

使用 Grep 工具在项目文档目录下对所有文档做全文关键词搜索：

**删除类关键词**：删除个人信息、删除隐私数据、删除个人数据、remove personal data、erase personal information、数据销毁、data erasure、数据擦除、clear personal data

#### 4.2 LLM 判断

对命中关键词的文档，使用 Read 工具读取相关段落，交给 LLM 判断：

- 是否描述了用户删除个人数据的方法或入口

**判定规则**：找到至少一个文档描述了删除个人数据的方法或入口即可通过。

#### 4.3 判定

**失败条件**：所有命中文档中均未描述删除个人数据的方法或入口。

**注意**：本步骤检查"有没有删除个人数据能力描述"，不深入验证机制的具体实现和有效性。

---

### 结果判定

| 角色 | PASS 条件 | FAILED 条件 |
|------|-----------|-------------|
| 数据控制者 | 三步校验均无失败项 | 任意一步出现失败项 |
| 设备供应者 | 至少找到一个文档描述了删除个人数据的方法 | 未找到任何删除个人数据的方法描述 |

## 前置条件

- 项目文档目录可访问，支持文件扫描和全文搜索
- 个人数据清单文件已就绪（Excel 格式，包含留存期列）
- 《个人数据留存规范》文件已就绪（数据控制者角色）
- 隐私声明文件已就绪（数据控制者角色）
- opencode内置工具可用（Glob、Grep、Read）

## 注意事项

1. 个人数据清单扫描到多个时，直接判定失败，无法确定使用哪个清单
2. 隐私声明扫描到多个时，直接判定失败，无法确定使用哪个声明
3. 留存期提取时需区分"未被留存的数据"和"被留存但未定义留存期的数据"，只有后者属于违规
4. 留存期比对需先完成数据类型匹配（清单条目 → 规范条目 → 声明条目），类型匹配不准会导致误判
5. 设备供应者分支检查"有没有删除个人数据能力描述"，不深入验证机制的具体实现和有效性
6. 关键词扫描可能遗漏使用同义词描述能力的文档段落，若关键词无命中可扩大关键词范围后再试
7. 数据类型匹配不准时，优先标记为WARNING而非FAILED，避免误判
8. 所有文件操作通过opencode内置工具（Glob、Grep、Read）完成，不使用bash

## 重要原则

**思考过程**：模型的分析和推理过程必须使用中文。

**实事求是**：
- 只报告真实文档中扫描和分析到的内容
- 如果文档中无相关内容，应明确告知用户"文档中未发现相关描述"
- 禁止编造留存期定义、处置机制或能力描述
- 禁止猜测文档中可能存在但实际未找到的内容

**工具优先**：
- 所有文件搜索必须通过opencode内置工具（Glob、Grep）完成
- 所有文件内容读取必须通过Read工具完成
- 不使用bash执行文件操作
- LLM分析基于Read工具读取的真实内容，不凭空推断

**保守判定**：
- 关键词扫描无命中时，可扩大关键词范围再试一次，仍无命中才判定失败
- 数据类型匹配不准时，优先标记为WARNING而非FAILED，避免误判

**角色分流**：
- 数据控制者严格执行步骤1-3，不执行步骤4
- 设备供应者仅执行步骤4，不执行步骤1-3
- 两种角色的判定标准和输出格式不同
